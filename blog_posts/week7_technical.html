SQL Injection... Should this scare you??

September 18, 2014

It should..


What is it...

Structured Query Language (SQL) Injection is one of the many web attack mechanisms to steal data from originations, suppliers, employees, and customers!! Really.. from YOU!!! It is one of the most common application attack techniques used today. It is an attack that takes advantage of improper coding of the web application that allows hackers to "inject" SQL commands into, say, login in forms to allow them access into your life! (or the database). It is so popular that when I did started my research I typed in SQL Injection and what do you think the one of the first 3 links were? Yeah t-shirts, there are even t-shirts for your kid...


Web applications (web apps) and Databases (db).
Db are a pretty much central to our everyday life on the web. They store data needed for web sites to deliver specific content to visitors and produce information to customers, suppliers, employees, (YOU!!!) or update, change information given by us, the user! Lets say the user visits a site that needs a user name and password. When the user submits their details a SQL query is generated and submitted to the data base for verification, once the information is verified, access is allowed! Since the site needs to be public, how else would the public access it, the security in place needs to allow public web traffic to communicate with the web app, the web app has open access to the db, it needs to change, update, add too, I'm sure the list goes on. SQL is the language the computer uses to perform those actions. In fact, as far as I know, SQL is the only way web apps to interact with database. Firewalls and other same type barriers wont work because the site needs to be public.


SQL Injection
The hacker uses SQL Injection to access the data bases. SQL Injection is the hacking technique in which attempts to pass SQL command statements though web applications for execution by the backend database. If not sanitized properly this will allow hackers to view information from the database OR even wipe it out. Using SQL Injection, the hacker can input specifically crafted SQL commands in hopes of by passing the login page barrier and see whats behind it. This is only possible if the inputs are not properly sanitized (made invulnerable) and sent directly with SQL query to the database. SQL Injection vulnerabilities provides the means for hacker to communicate directly to the data base. In 2012 a study was done that showed on average web application received 4 attack campaigns per month and retailers received twice as many attacks.. DUH...


Example

Here is a common application used to verify user login

Username:


Password:


SQL query
1
SELECT * FROM users WHERE Username='$username' AND Password='$password'



This is designed to show all records from the table user for a username and password supplied by a user. Using a web interface when prompted, a malicious user might enter"

1' or '1' = '1

Resulting in the query
1
SELECT * FROM Users WHERE Username='1' OR '1' = '1' AND Password='1' OR '1' = '1'

The hacker has effectively injected a whole OR condition into the authentication process. Worse, the condition '1' = '1' is always true, so this SQL query will always result in the in the authentication process being bypassed. Using characters like ";" to append another query on the end of an existing one, and -- to comment out(and therefore cut off) a part of an existing query, a hacker could potentially delete entire tables, or change the data they contain. The consequences of a SQL injection attach could be:

  • Loss of data confidentiality
  • Loss of data integrity
  • Loss of data
  • Compromise of the entire network

Conclusion

I want to be a HACKER when I grow up! As newbs, how can we combat this issue? Learn how it done! The only way to destroy something is to know how it was created! So it seems that SQL injections thrive on poorly written code. As 'newbs' we really need to focus on security when dealing with applications. Assume all user-submitted data is evil, validate and sanitize everything. Don't use dynamic SQL when it can be avoided. Get rid of any database functionality that you don't need to prevent a hacker from taking advantage of it.
